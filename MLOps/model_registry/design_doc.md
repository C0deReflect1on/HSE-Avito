Design Doc: Model Registry


## Какую проблему мы решаем

Существующий способ хранения моделей имеет такой вид:

```
models/
models/mlds_1/my_model_v1/
models/mlds_3/sft_modell_v_123/
models/mlds_180/model_v1_v0_with_rank_dataset_0/
models/mlds_180/model_v1_v0_with_rank_dataset_1/
models/mlds_180/model_v1_v0_without_rank_dataset_0/
models/mlds_180/model_v1_v0_without_rank_dataset_1/
```

Это представляет собой хаос по следующим причинам:
Хранение организованно в виде вложенных папок + длинных названий моделей по итогу, если не хватило глубины папки.
Из-за этого смешивается в кучу названия моделей, датасетов на которых обучали, набор фичей и другие параметры.

В совокупности это приводит как к проблемам распознования, чем модели принципиально друг от друга отличаются (долгие названия тяжело читать),
повышается риск ручной ошибки (в описании названия конфигурации) + трудность с поиском по различным срезам, так-как структурные уровни не являются хранилищем.


## Цели

Что хотим:

1. Гарантированное конфигурирование модели и её зависимостей
2. Поиск по параметрам моделей, например найти все модели обученные на одном и том же наборе фичей/датасете за тот же изначальный период времени
3. Понятный флоу воспроизведения результатов (что нужно сделать, не будет гарантии что это получится, но будет четко видно что является блокером)
4. Возможность выгрузки бинарника модели через конфиг на pod сервиса

Что не хотим:

1. Не хотим обеспечить полный цикл тренировки моделей
2. Не хотим обеспечить переобучение моделей на уже готовых данных
3. Обеспечить real-time инференс модели из модель реджистри
4. Не упарываемся в масштабирование на этом этапе — решаем задачу систематизации, а не высоконагруженной раздачи моделей


## Риски

1. Если нет feature стора, зависимость на датасет/версии/мета информацию пробросить существенно тяжелее — будет точкой возникновения ошибок и расхождений
2. Может захотеться расширение в сторону поддержки цикла разработки моделей (подтаскивать результаты экспериментов — хотя бы train/val/test метрики). Пока это out of scope, но структуру стоит закладывать с учетом этого
3. Может поменяться понимание того как хотим хранить модели с точки зрения lineage (наследования) — условно что вот эти модели являются тюнингом вот этой базовой, или у нас есть bert и вот все поисковые берты которые мы из него сделали. Это потребует изменения структуры, но сейчас сознательно не закладываем


## Требования

### Функциональные

Что система должна уметь делать:

1. **Регистрация модели** — загрузить бинарник модели + её мета-информацию (название, тип, датасет, фичи, гипер-параметры, владелец, ссылка на код) одной операцией. Нельзя зарегистрировать модель без обязательных полей — система должна валидировать
2. **Версионирование** — одна и та же модель может иметь несколько версий. Каждая версия — отдельная запись со своим бинарником. Старые версии не удаляются и не перезаписываются
3. **Поиск и фильтрация** — найти модели по произвольной комбинации параметров: model_type, dataset, feature_names, owner и тд. Это основной сценарий использования для MLE
4. **Скачивание модели** — получить бинарник конкретной версии модели, или latest версию по имени. Это нужно как для ручной работы, так и для автоматической выгрузки на pod
5. **Получение мета-информации** — посмотреть все параметры модели без скачивания бинарника (чтобы понять, что это за модель, на чем обучена, кто владелец)
6. **Управление стадиями** — модель проходит через стадии `experimental` → `production` → `archived`. Перевод между стадиями — явное действие. `latest` по умолчанию отдает последнюю `production` версию, чтобы экспериментальные модели не попадали на поды автоматически

### Нефункциональные

Каким свойствам система должна удовлетворять:

1. **Доступность хранилища** — бинарники моделей должны быть доступны всегда, включая момент выкатки нового релиза. Если новая версия модели сломалась, нужно уметь откатиться на предыдущую — и она должна быть доступна
2. **Консистентность мета-информации** — если модель зарегистрирована, то её метадата и бинарник существуют оба. Не должно быть ситуации когда запись в БД есть, а файл в S3 потерялся (или наоборот)
3. **Идемпотентность регистрации** — повторная регистрация той же модели+версии не должна создавать дубликаты или ломать существующую запись
4. **Изоляция от деталей хранения** — потребители работают только через API, не знают про S3 пути и структуру БД напрямую. Это позволяет менять реализацию под капотом
5. **Нагрузка** — на текущем этапе не оптимизируем под высокую нагрузку. Ожидаемый профиль: единицы-десятки запросов в минуту на поиск, редкие (единицы в день) регистрации новых моделей, всплески скачиваний при деплоях (решается через presigned URL, чтобы не нагружать API)


## Архитектура (высокоуровнево)

Система состоит из трех компонентов:

```
┌─────────────┐       ┌──────────────┐       ┌─────────────┐
│  API слой   │──────▶│  PostgreSQL  │       │     S3      │
│  (REST)     │──────▶│  (метадата)  │       │  (бинарники)│
└─────────────┘       └──────────────┘       └─────────────┘
      │                                             ▲
      │           upload/download моделей           │
      └─────────────────────────────────────────────┘
```

**PostgreSQL** — хранит мета-информацию о моделях.
**S3** — хранит сами бинарники моделей.
**API** — единая точка входа для регистрации, поиска и выгрузки моделей.

Почему так:
- Моделей будет немного, нагрузка на метадату низкая — PostgreSQL за глаза хватает, плюс он дает нормальную фильтрацию/поиск из коробки
- S3 дает надежное key-value хранение бинарников, версионирование, не надо думать про downtime хранилища
- API слой изолирует потребителей от деталей хранения: сегодня S3, завтра что-то другое — снаружи ничего не меняется


## Реализация

### Мета-информация о модели (PostgreSQL)

Хотим чтобы модель у нас описывалась как сущность со следующими полями:

| Поле           | Описание                                                                                  |
|----------------|-------------------------------------------------------------------------------------------|
| `id`           | Внутренний id (PK)                                                                        |
| `name`         | Название модели                                                                           |
| `version`      | Версия модели                                                                             |
| `model_type`   | Класс модели (catboost, logreg, bert, CLIP, DSSM etc)                                     |
| `dataset`      | Ссылка на датасет — hdfs/yt/etc, или на сущность feature_store (что более желательно)     |
| `params`       | Гипер-параметры обучения (jsonb)                                                          |
| `feature_names`| Названия фичей (jsonb)                                                                    |
| `s3_path`      | Путь к бинарнику модели в S3                                                              |
| `git_path`     | Путь к коду,артефакту обучения модели                                                     |
| `stage`        | Стадия жизненного цикла: `experimental` / `production` / `archived`                       |
| `created_at`   | Время загрузки                                                                            |
| `owner`        | Владелец (abc группа, ~mtканал и тд, короче то, что поможет выйти на владельцев  )        |

`params` и `feature_names` храним как jsonb — это позволяет делать фильтрацию по произвольным полям прямо в PostgreSQL, не усложняя схему.

**Зачем `stage`:**
Нужно отличать экспериментальные модели от тех, что реально крутятся на проде. Без этого `latest` опасен — кто-то залил эксперимент и он стал "последней версией" для всех подов.

Жизненный цикл модели простой:
- `experimental` — модель только загружена, ещё не проверена. Это дефолт при регистрации
- `production` — модель проверена и готова к использованию в сервисах
- `archived` — модель больше не используется, но не удаляем (для воспроизводимости и отката)

Перевод между стадиями — явное действие через API (`PATCH /models/{id}/stage`), не автоматическое.

### Хранение бинарников моделей (S3)

S3 используется как key-value хранилище для бинарников.

Ключевые моменты:
- Путь в S3 формируется детерминированно из `name` + `version` (например `models/{name}/{version}/model.bin`)
- Версионирование на уровне нашей таблицы (не S3 versioning) — проще контролировать и не зависим от фичей конкретного S3
- Если выкатили новый релиз и нужно откатиться — старая версия всегда доступна по своему пути, ничего не перезаписывается

### API

REST API является единой точкой входа. Базовые ручки:

- `POST /models` — зарегистрировать новую модель (метадата + загрузка бинарника). Модель создается со `stage=experimental`
- `GET /models` — поиск по параметрам (model_type, dataset, feature_names, stage и тд)
- `GET /models/{id}` — получить метадату конкретной модели
- `GET /models/{id}/download` — скачать бинарник модели
- `GET /models/{name}/latest` — последняя `production` версия модели по имени. Опционально `?stage=experimental` если нужна последняя экспериментальная
- `PATCH /models/{id}/stage` — перевести модель между стадиями (experimental → production → archived)

Для выгрузки модели на pod сервиса: pod через API запрашивает нужную версию модели (или latest), API отдает presigned URL на S3, pod скачивает напрямую из S3. Так API не становится бутылочным горлышком при раздаче тяжелых файлов.

То есть типичный флоу такой: MLE загружает модель → она `experimental` → проверили метрики, всё ок → `PATCH` в `production` → pod через `GET /models/musicRec/latest` получает именно эту версию, а не чей-то вчерашний эксперимент.
