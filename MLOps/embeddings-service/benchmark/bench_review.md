# Benchmark Review

## Что сравнивалось

Сравнивались 4 режима:

- `hf_no_batch`
- `hf_batch_bw500`
- `onnx_no_batch`
- `onnx_batch_bw500`

Во всех файлах по 5 прогонов на сценарий. Ниже выводы сделаны по медианам.

Важно: `no_batch` в этих результатах означает только отключение серверного dynamic batching в `InferenceService`. Это не означает, что в запросе всегда был один текст: сам `run_benchmark.py` все равно гоняет сценарии с `batch=1/8/16/32`.

## Короткий вывод

Основной победитель по скорости в этих замерах: `onnx_no_batch`.

- У него лучшая latency почти во всех ключевых сценариях.
- У него лучший `texts/s` почти во всех ключевых сценариях.
- Его главный минус: заметно более высокий `RAM max`, особенно на высоком `concurrency`.

Dynamic batching с `BATCH_WINDOW_MS=500` в этом прогоне почти везде ухудшил и latency, и throughput как для `hf`, так и для `onnx`. Единственный заметный плюс batching здесь: снижение пикового RAM, особенно у `onnx`.

## Подтвержденные выводы

### 1. `onnx_no_batch` быстрее `hf_no_batch`

По медианам:

| Scenario | HF no batch | ONNX no batch | Что видно |
|---|---:|---:|---|
| `baseline` p50 | `272 ms` | `154 ms` | ONNX быстрее примерно на `43%` |
| `baseline` texts/s | `94` | `166` | ONNX выше примерно на `77%` |
| `batch_sweep`, `batch=8` texts/s | `210` | `270` | ONNX выше примерно на `29%` |
| `conc_sweep`, `batch=8`, `conc=16` texts/s | `233` | `296` | ONNX выше примерно на `27%` |
| `sustained` texts/s | `242` | `280` | ONNX выше примерно на `16%` |

Вывод: если цель именно максимальная производительность, по этим данным `onnx_no_batch` лучше `hf_no_batch`.

### 2. Dynamic batching с окном `500 ms` почти везде повредил производительности

Для `hf`:

- `baseline`: `94 -> 46 texts/s`, p50 `272 -> 554 ms`
- `conc_sweep`, `batch=8`, `conc=16`: `233 -> 186 texts/s`, p50 `1589 -> 1816 ms`
- `sustained`: `242 -> 186 texts/s`, p50 `10201 -> 12912 ms`

Для `onnx`:

- `baseline`: `166 -> 56 texts/s`, p50 `154 -> 457 ms`
- `conc_sweep`, `batch=8`, `conc=16`: `296 -> 221 texts/s`, p50 `1200 -> 1535 ms`
- `sustained`: `280 -> 223 texts/s`, p50 `8803 -> 10784 ms`

Вывод: при `BATCH_WINDOW_MS=500` серверное batching не ускорило систему, а добавило ожидание.

### 3. Dynamic batching дал выигрыш в памяти, но ценой скорости

Самый заметный эффект у `onnx`:

- `conc_sweep`, `batch=8`, `conc=32`: `1203 MB -> 558 MB`
- `sustained`: `1330 MB -> 560 MB`

У `hf` снижение памяти тоже видно, но эффект слабее:

- `conc_sweep`, `batch=8`, `conc=16`: `620 MB -> 424 MB`
- `sustained`: `709 MB -> 427 MB`

Вывод: batching с окном `500 ms` можно рассматривать как компромиссный режим для экономии RAM, но не как режим для максимального throughput.

### 4. Внешний batching на уровне запроса уже сам по себе работает

Даже без серверного dynamic batching `run_benchmark.py` уже посылает запросы с несколькими текстами в одном запросе.

Например для `onnx_no_batch`:

- `batch=1`, `conc=4`: `280 texts/s`
- `batch=8`, `conc=4`: `270 texts/s`
- `batch=16`, `conc=4`: `227 texts/s`
- `batch=32`, `conc=4`: `226 texts/s`

Для `hf_no_batch` картина похожая:

- `batch=1`, `conc=4`: `109 texts/s`
- `batch=8`, `conc=4`: `210 texts/s`
- `batch=16`, `conc=4`: `198 texts/s`
- `batch=32`, `conc=4`: `187 texts/s`

Вывод:

- у `hf` полезный рост есть при переходе от `batch=1` к `batch=8`
- дальнейшее увеличение batch до `16/32` уже не дает роста `texts/s`
- у `onnx` лучший режим в этих замерах вообще не `batch_sweep`, а `conc_sweep` с `batch=8`

## Почему выводы именно такие

Причина не только в цифрах, но и в логике кода.

`run_benchmark.py` уже сам формирует payload с несколькими текстами через `make_payload(batch)`, а в сценариях явно перебираются `batch=1/8/16/32`. То есть клиентский batching уже присутствует даже в файлах `*_no_batch.csv`.

Дополнительно `InferenceService` при `BATCHING_ENABLED=true` складывает запросы в очередь и ждет до заполнения батча или до истечения `batch_window_ms`. При `BATCH_WINDOW_MS=500` это большое окно ожидания для latency-sensitive нагрузки. Поэтому в этих замерах серверный dynamic batching, скорее всего, добавил больше времени ожидания, чем сэкономил на объединении запросов.

Именно поэтому я формулирую вывод не как "`batching` плохой всегда", а как:

- batching с окном `500 ms` оказался неудачным именно в этом профиле нагрузки;
- ONNX здесь лучше HF по скорости;
- batching здесь полезен в основном как способ уменьшить пиковую память, а не увеличить производительность.

## Практический итог

Если выбирать режим по этим CSV:

- для максимальной скорости: `onnx_no_batch`
- для более умеренного потребления памяти: `onnx_batch_bw500` или `hf_batch_bw500`, но с ожидаемой потерей latency и throughput
- для следующего шага: отдельно прогнать `onnx` и `hf` с меньшим окном, например `10-50 ms`, потому что текущее окно `500 ms` слишком велико для честной оценки dynamic batching
